Background

Network Time Protocol (NTP) is a protocol that is used to synchronize computer clock times in a network of computers. Developed by David Mills at the University of Delaware, NTP is now an Internet standard. NTP uses Coordinated Universal Time (UTC) to synchronize computer clock times to a millisecond, and sometimes to a fraction of a millisecond. Accurate time across a network is important for many reasons. Even small fractions of a second can cause problems. For example, distributed procedures depend on coordinated times to ensure that proper sequences are followed. Security mechanisms depend on coordinated times across the network. File system updates carried out by a number of computers also depend on synchronized clock times. Air traffic control systems provide a graphic illustration of the need for coordinated times, since flight paths require very precise timing (imagine the situation if air traffic controller computer clock times were not synchronized).

Our Approach  -  Introduction

Traditional NTP works on the Request-Reply model. A client requests to synchronize its clock with the server. The server responds by replying to the request.In the Figure1, the approach followed by traditional NTP is shown.Stratum1 or the client sends a request to Stratum0(server) to synchronize its clock.Stratum0 sends a reply to the request of Stratum1.

As shown in the Figure1 send and receive timestamps T0 , T1 , T2 , T3 are recorded at the server and client. Using these timestamp values traditional NTP calculates the offset as:

offset   =  ( T1  +  T2 )  / 2  -  ( T0  +  T3 )  / 2

While calculating the offset NTP asumes that the value of Tin and Toutare the same. Which is not true.

Our approach is that we calculate the value of Tin. As shown in Figure2. Stratum0 sends a packet to Stratum1.Send and receive timestamps are recorded at Stratum0 and Stratum1 respectively and the value of offset can be calculated as:

offset   =  T1  - ( T0  +  Tin )

Related Works

Paper [7] talks about another time synchronization protocol known as Precision Time Protocol (PTP) , the transmission delay is calculated by sending a FOLLOW UP message immediately after SYNC message. Also the propagation delay is calculated by sending DELAY REQUEST and DELAY RESPONSE messages. Thus, the offset value on a direct link is calculated by using propagation and transmission delay. To compensate for the wait times and/or jitter when passing through routers/switches, intermediate synchronization in a point to point connection takes place. A clock connected to switch/router is synchronized to a directly connected reference clock, which in turn, behaves as a reference clock at every other interface and synchronizes the connected nodes. Unlike PTP [7], we do not synchronize the clock on every router / switch on a direct link or on a point to point connection rather we calculate end-to-end transit time between local and remote server.TPSN (Timing-sync Protocol for Sensor Networks) [2] is one of the effective protocols proposed to synchronize sensor networks. In TPSN Pair-wise synchronization is done by timestamping the Synchronization pulse packets and Synchronization acknowledgement packets between a pair of nodes. Timestamping is done at the lowest possible network layer.
