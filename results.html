a.How Traditional NTP would fail?




To demonstrate how traditional NTP fails, let us consider a scenario that the two clocks of S0 and S1 are offset by 10 time units. Referring to Figure 7, consider S1 sends an NTP packet to S0 at time T0 = 10 time unit, the time taken by this packet to reach S0 be TTX = 5 time unit. Therefore,

TR  = T0 + TTx + offset  =  10+5+10  =  25timeunit.

Let's say S0 sends a reply after 10 time unit at TT = 35 time unit. Time taken for the packet to reach from S0 to S1, TRX =10 time unit, so packet will reach S1 at TI.

TI = TT - offset + TRX  =  35 - 10 + 10  =  35timeunit.

We find offset using below equations :-

TLocal  =  ( To  +  TI )  /  2  =  ( 10  +  35 )  /  2  =  22.5  timeunit.

TRemote  =  ( TR  +  TT )  /  2  =  ( 25  +  35 )  /  2  =  30  timeunit.

Offset  =  TRemote  -  TLocal  =  30  -  22.5  =  7.5  timeunit.

From the above calculations we can see that the error in offset is  10  -  7.5  =  2.5  timeunit. This error is nothing but the half of the difference in TTx and TRX. So we can say that traditional NTP would give results with an error of  ( TTx  -  TRX )  /  2. This error is completely eliminated in our implementation as our method does not depend on the premise that incoming and outgoing transit times are equal, but we measure the one way transit time. b.Our Results




The above plot is is for offset in milliseconds with respect to UTC epoch time for four scenarios. The four scenarios are legacy NTP over asymetric link and direct link, custom NTP over link with routers and direct link. To compare the our implementation with that of the legacy NTP we measured the offset between master and slave with custom NTP and legacy NTP simulatenously.To compare the results of the four scenarios with each other, we do not update the slave clock and only record the offset. Because of this we see that the graph is not a straight line but with a slope. The slope of the line is nothing but the clock drift rate of the slave. The difference between the transit times in case of asymetric NTP was 129micro seconds when we measured these set of values. Now refer to the second graph which is zoomed version of the first graph. If we consider the points where the grpahs intersect with Y axis we can see that the legacy NTP over asymetric link does not match with that of legacy NTP over direct link. This clearly shows the problem with NTP. The difference between the offset values is 100micro seconds. It is seen that the offset values for custom NTP in both the scenarios and that of legacy NTP over direct link coincide, the graph are overlapping for entire range with very little deviation. This proves two points:

  1. Our implementation is independent of the path taken by the packet.
  2. The offset that we get is correct as this coincides with that of legacy NTP over direct link. Legacy NTP is a code that has been accepted and used world wide. Also if we run it over direct link we know that the incoming and and outgoing transit times are same as transmission and reception is over same link. So we can consider it as reference.
